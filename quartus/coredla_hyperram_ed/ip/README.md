# Track M — CoreDLA AGX3 IP + compiled .aot for the 4 MLPerf Tiny models

Goal (see task brief / `docs/tinyml_all_models_plan.md`, `docs/ph3_coredla_nextsteps.md`): produce a
single, model-agnostic CoreDLA IP built against `models/arch/AGX3_Performance.arch` (the
external-memory/DDR-streaming AGX3 arch — the only AI Suite 2026.1.1 AGX3 file that accepts an
INT8/FakeQuantize graph at all, see `models/arch/README.md`), plus one compiled `.aot` per MLPerf
Tiny model, ready for the HyperRAM-backed on-board run (Track B / the orchestrator's job).

This dir is gitignored build output (`quartus/*/ip*` conventions elsewhere in this repo); nothing
here is committed to git.

## 1. The CoreDLA IP (`coredla_ip/`)

`coredla_ip/altera_ai_ip/verilog/AGX3_Performance_AGX3/` — generated by `dla_create_ip --arch
models/arch/AGX3_Performance.arch` (external-memory/DDR config, **not** DDR-free — DDR-free has no
AGX3 arch variant in this AI Suite release, see `models/arch/README.md` finding #2). This is a
**copy of the already-generated, already-verified** IP at `_ph3_ed_hyperram/coredla_ip/` (PH3
hardware-bringup track, HyperRAM-backed CoreDLA platform) — copied here rather than regenerated
because `dla_create_ip` for a streaming/external-memory arch is **model-agnostic** (weights stream
in at runtime, nothing is baked into on-chip ROMs the way a DDR-free arch would), so the same IP
serves all four models below; regenerating it would reproduce byte-identical RTL at real Docker/EDA
cost for no benefit. Confirmed from `interface_param.tcl` / `default_param_values.tcl` inside the
IP:

| Parameter | Value |
|---|---|
| `ARCH_C_DDR_AXI_DATA_WIDTH` | **256** (bits) — matches the task brief's expectation |
| `ARCH_C_DDR_AXI_ADDR_WIDTH` | 32 |
| `ARCH_C_DDR_AXI_READ_ID_WIDTH` / `WRITE_ID_WIDTH` | 2 / 5 |
| `C_CSR_AXI_ADDR_WIDTH` / `C_CSR_AXI_DATA_WIDTH` | 11 / 32 |
| CSR base address | `0x8000_0000` (resolved handshake, see `docs/coredla_csr_handshake_findings.md`) |
| Arch long-form name | `kvec16xcvec16_i12x1_fp12agx_sb8192_xbark16_actk16_poolk4` (16x16 PE array, `stream_buffer_depth=8192`) |

This 256-bit AXI4 DDR port is what `rtl/hyperbus/axc3000_hyperram_axi4.sv` /
`axi4_hbmc_bridge.sv` (this repo, see top-level task brief) bridge to the HyperRAM PHY.

## 2. Per-model compile summary

All four compiled **100% to FPGA** (`dla_compiler --march models/arch/AGX3_Performance.arch
--fplugin HETERO:FPGA`, exit 0, `model_analyzer_report.txt`: "All layers are supported" — no CPU
fallback is even possible for a quantized/FakeQuantize graph under HETERO:FPGA, see
`docs/toolchain.md` / `results/reports/ph0_estimator.md`, so a clean exit *is* the 100%-FPGA proof).
FPS below is a **fresh, real `dla_compiler --fanalyze-performance` run** at
`--fassumed-memory-bandwidth 342` (MB/s) — the *measured* AXC3000 HyperRAM ceiling
(`results/reports/hyperbus_bw.md`, memory note "HyperRAM IP + measured bandwidth"), not the 200-400
MB/s PH0 sweep points and not the unrealistic 6400 MB/s "per IP instance" default. `kind: estimate`
per AGENTS.md (it's `dla_compiler`'s own performance-estimator mode, not a hardware measurement) —
all 4 are memory-bandwidth-bound at 342 MB/s (`Memory Bandwidth Bottleneck Detected` in every
report).

| Model | Compiles? | `.aot` | `.aot` size | FPS @ 342 MB/s | DDR filter (weight) reads/inf | DDR config bytes/inf | DDR feature spill/inf | Total DDR traffic/inf |
|---|---|---|---|---|---|---|---|---|
| `ad-toycar` | **yes** (no rewrite needed) | `models/ad-toycar/ad-toycar.aot` | 1,683,968 B | **748.41** | 0.53 MB | 0.01 MB | 0.00 MB | 0.54 MB |
| `ds-cnn-kws` | **yes** (rewritten) | `models/ds-cnn-kws/ds-cnn-kws.aot` | 1,777,099 B | **257.20** | 1.47 MB | 0.03 MB | 0.04 MB | 1.54 MB |
| `resnet8-cifar10` | **yes** (rewritten) | `models/resnet8-cifar10/resnet8-cifar10.aot` | 611,702 B | **1501.88** | 0.18 MB | 0.02 MB | 0.03 MB | 0.23 MB |
| `mobilenetv1-025-vww` | **yes** (rewritten) | `models/mobilenetv1-025-vww/mobilenetv1-025-vww.aot` | 1,984,503 B | **180.34** | 0.83 MB | 0.07 MB | 1.05 MB | 1.95 MB |

"DDR feature spill" = feature reads + feature writes (activation traffic through external memory,
not counting weights/config). All bytes are **per single inference**, at 1x I/O buffering (the
report's own caveat: "COREDLA RUNTIME DEFAULTS TO 5" buffers, so real steady-state HyperRAM traffic
will be higher — a runtime/DMA detail, not this report's to resolve).

Raw `dla_compiler` invocation (each model, `MODEL_XML` = the int8 IR listed in §3):
```
source scripts/env.sh
dla_compiler --fanalyze-performance \
  --march models/arch/AGX3_Performance.arch \
  --network-file <MODEL_XML> \
  --foutput-format open_vino_hetero --fplugin HETERO:FPGA \
  --o out.aot --dumpdir <dumpdir> --overwrite-output-files \
  --fdump-performance-report perf.txt --fassumed-memory-bandwidth 342
```
Full logs/reports: `models/compiled/track_m/<model>/compile_log.txt` +
`.../tf2onnx/reports/perf_0.txt` (gitignored scratch; this dir's `perf_342mbps.txt` /
`model_analyzer_report.txt` per model are the durable copies).

## 3. Where each model's CoreDLA-compilable INT8 IR came from

`ad-toycar` compiles as-is (`models/ir/ad-toycar/int8/`, no rewrite — PLAN's only model that was
never blocked, see `results/reports/ph0_estimator.md`). The other three all failed pre-rewrite with
the exact `dla_compiler` errors in `results/reports/ph0_sweep_attempts.csv` (re-confirmed fresh
against each model's *baseline* IR before trusting the rewrite — see the `baseline`/`_baseline`
compile dirs referenced below, each showing the identical unsupported-layer name from the CSV:
`Constant_16879` resnet8, `Constant_4649` vww, the AvgPool node ds-cnn).

| Model | Real blocker (fresh `dla_compiler` read, not just the CSV prose) | Fix applied | Equivalence check |
|---|---|---|---|
| `ds-cnn-kws` | Final `AveragePool` over the full 25x5 time/freq grid exceeds the arch's pool ceiling (window 3, stride 4) | `sw/model_prep/graph_ops.make_coredla_friendly` (`pool_decompose`, `--pool-strategy conv` chosen automatically) — **used exactly as documented**, no other change needed; `check_coredla_friendly` reports `remaining_issues: []` | onnxruntime, original vs. rewritten ONNX, 8 random inputs at the real `[1,49,10,1]` shape: **max abs diff = 1.19e-7** (fp32 rounding noise) |
| `resnet8-cifar10` | (a) global `AveragePool` 8x8 exceeds the same pool ceiling — `pool_decompose` (cascade strategy) fixes this alone (`models/scratch/ir/resnet8_pool/`); (b) **a second, separate blocker survives pool-fix-only** (`resnet8_pool_compile/` still fails, `Constant_23` unsupported) — the OpenVINO IR conversion of an NHWC-native ONNX graph auto-inserts a `Transpose` ahead of the flatten→`MatMul` head that CoreDLA rejects. This `Transpose` does **not** exist in the source ONNX (`sw/model_prep/graph_ops.fold_transposes` correctly finds nothing to fold — it's inserted downstream by `ovc`), so `graph_ops`'s static ONNX-level check cannot see it; matches `docs/tinyml_all_models_plan.md`'s explicit warning that the CSV prose and the real blocker can disagree and must be re-derived fresh | `pool_decompose` (cascade) **+** re-exporting the whole graph NCHW-native (`models/scratch/resnet8_pool_nchw.onnx`) so `ovc` never needs to insert that Transpose in the first place — mathematically identical to `transpose_fold`'s intent (removing a placement-only permutation) but applied by avoiding its insertion rather than folding it after the fact | onnxruntime, original (NHWC input) vs. rewritten (NCHW input, same data transposed) ONNX, 8 random inputs at `[1,32,32,3]`/`[1,3,32,32]`: **max abs diff = 1.19e-7** |
| `mobilenetv1-025-vww` | A **leading** `Transpose([0,3,1,2])` converting the graph's NHWC input to NCHW before the first conv — not a Transpose-before-FC case (`graph_ops.fold_transposes` correctly reports nothing to do here either: it only targets Transpose immediately preceding Gemm/MatMul), but `dla_compiler` rejects it as an unsupported Constant/placement node (`Constant_4649`, matching the CSV) | Remove the leading Transpose and make the graph's own input NCHW (`models/scratch_vww/mobilenetv1-025-vww-nchw.onnx`) — an exact input-layout change, not a numerical one | onnxruntime, original (NHWC input) vs. rewritten (NCHW input, same data transposed) ONNX, 8 random inputs at `[1,96,96,3]`/`[1,3,96,96]`: **max abs diff = 0.0** (bit-exact) |

**Honest caveat on `ds-cnn-kws`/`resnet8-cifar10`/`mobilenetv1-025-vww` rewrite provenance:** the
NCHW re-export step for resnet8/vww and the NNCF INT8 requant for all three were done in an earlier
session's scratch directories (`models/scratch/`, `models/scratch_vww/`, `scratch/dscnn/` —
gitignored, not committed anywhere) rather than freshly re-run end-to-end by this task. What *this*
task independently re-verified fresh, right now: (1) `graph_ops.make_coredla_friendly` really does
fully resolve `ds-cnn-kws` on its own (ran it live, `remaining_issues: []`); (2) the onnxruntime
equivalence numbers in the table above (re-ran all three, did not just copy old printed numbers);
(3) the actual `dla_compiler --fanalyze-performance` compile of all four models' int8 IR against
`AGX3_Performance.arch` at the realistic 342 MB/s bandwidth, fresh, capturing the `.aot` + reports
now living in this directory. The INT8 IR themselves (`models/*/int8/*.xml,*.bin` referenced below)
were NNCF-quantized using each model's real calibration split
(`sw/model_prep/quantize_int8.py`'s per-model `calibration_samples()`), evidenced by
`scratch/dscnn/quant_s1.log` (`INT8_OK 300` — 300-sample calibration, matching the pipeline's
documented budget) and `scratch/dscnn/acc.log` (`N=800 transformed_int8_acc=0.9075
reference_int8_acc=0.9100 argmax_agreement=0.9975` — INT8 accuracy after the rewrite is within 0.25
points of the reference INT8 model, i.e. the rewrite did not cost accuracy). No equivalent saved
accuracy log was found for resnet8/vww's INT8 requant in this session's scratch state; their
NCHW-rewrite fp32-equivalence (table above) is independently re-verified, but their post-NNCF INT8
accuracy number should be re-run through `sw/model_prep/eval_int8_cpu.py` before treating it as a
closed accuracy-gate checkbox (Phase A2 in `docs/tinyml_all_models_plan.md`) — flagging rather than
asserting it.

Rewritten ONNX + int8 IR + full model_analyzer/perf reports are copied alongside each `.aot` in
`models/<model>/`.

## 5. Handoff to Track B / the on-board run

- Load `coredla_ip/` into the HyperRAM-backed platform (`_ph3_ed_hyperram/` /
  `docs/ph3_integration.md`'s EMIF→HyperRAM swap, already applied there) — same IP, no per-model
  rebuild needed.
- Each model's `.aot` is what the CoreDLA runtime loads at inference time; the weight/config bytes
  in §2's table are what must be resident in / streamed from HyperRAM per inference.
- CSR base `0x8000_0000` (`sw/host/coredla_csr_handshake.py`) is unchanged — this IP generation
  didn't touch the CSR map.
- Not verified here (needs the physical board, out of this task's scope): actual on-device latency,
  accuracy-on-device, and whether the runtime's 5x I/O buffering default changes the steady-state
  HyperRAM bandwidth picture from the 1x-buffer numbers in §2.
