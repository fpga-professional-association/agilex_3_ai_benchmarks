# MLPerf Tiny reference bundles — format, byte layout, and how the orchestrator uses them

Deliverable for the on-board benchmark prep (PLAN §9 PH3 / `docs/onboard_benchmark_plan.md`): a
per-model reference bundle of real-test-set INT8 input tensors + the CPU-INT8 reference prediction
for each, so the orchestrator can parity-gate and score every on-board CoreDLA run **without
touching the board itself here**. All four MLPerf Tiny models are ready:
`ad-toycar`, `ds-cnn-kws`, `resnet8-cifar10`, `mobilenetv1-025-vww`.

Generator: `sw/model_prep/build_tiny_bundle.py` (run from `sw/model_prep/`). Loader (already existed,
unit-tested, unchanged by this work): `sw/host/run_tiny_benchmark.py:load_bundle`.

## 0. tl;dr for the orchestrator

```
bundle = run_tiny_benchmark.load_bundle("results/tiny_bundles/<model_id>")
# bundle.records[i].input_bytes -- raw INT8 bytes, engine-native layout, ready to DMA/JTAG-write
# bundle.records[i].label       -- ground truth class (classifiers) / file anomaly flag (AD)
# bundle.records[i].cpu_pred    -- CPU-INT8 (deployed IR) argmax, classifiers only
# bundle.input_addr / output_addr / output_bytes -- from reference.json (defaults if unset)
```
Feed each `input_bytes` at the model's config-determined input address (HyperRAM-guard-banded per
`docs/coredla_hyperram_onboard_findings.md`, or the DDR-free streaming path), run one inference,
argmax the INT8 output (`run_tiny_benchmark.argmax_int8`), and compare against `cpu_pred`/`label`.
`sw/host/run_tiny_benchmark.py --mode both` already does all of this end-to-end once given
`--bundle results/tiny_bundles/<model_id>`.

## 1. Bundle layout on disk

```
results/tiny_bundles/<model_id>/
  reference.json          # loader-consumed: model_id, metric, output_bytes, records[]
  manifest.json            # human/tooling doc: shapes, dtype, layout, quant scale, provenance
  records/rec_00000.int8   # raw INT8 input tensor, engine-native layout, N bytes (no padding/header)
  ...
  accuracy_full/           # GITIGNORED — larger record set, same format, regenerate on demand
    reference.json
    records/rec_00000.int8
    ...
```
`reference.json` is **exactly** the format `run_tiny_benchmark.load_bundle` already expected
(`docs/tiny_hardware_benchmark_runbook.md` §0.3) — this work populates it, it doesn't change it.
Extra per-record keys beyond what the loader reads (e.g. AD's `audio_file`/`cpu_recon_mse`) are
harmless — `load_bundle` only reads named keys via `.get()`.

The top-level `results/tiny_bundles/<model_id>/` directory (16 records, tens–hundreds of KB, see
table below) is small enough to commit directly and is **not** gitignored. `accuracy_full/` (a
larger, still-not-the-full-10000/4890-record packaged set — see §4) **is** gitignored
(`.gitignore`: `results/tiny_bundles/*/accuracy_full/`) and is regenerated by re-running the builder
— never hand-maintained.

| model | records (committed / accuracy_full) | bytes/input | committed dir size |
|---|---|---|---|
| `ad-toycar` | 16 / 256 | 640 | 80 KB |
| `ds-cnn-kws` | 16 / 256 | 490 | 80 KB |
| `resnet8-cifar10` | 16 / 256 | 3,072 | 80 KB |
| `mobilenetv1-025-vww` | 16 / 256 | 27,648 | 464 KB |

## 2. The exact input byte layout CoreDLA expects — and how this was determined

This is the open item `docs/record_format.md` and `docs/parity_debugging.md` (#2, "systematic class
skew → layout transpose") flag as unresolved until "the `dla_compiler` compile report" is read.
Resolved here by **directly inspecting the ONNX graph that was actually fed to `dla_compiler`** —
`quartus/coredla_hyperram_ed/ip/models/<model_id>/<model_id>-rewritten.onnx` (the file whose compile
produced the `.aot` on the board) — rather than trusting `models/ir/<id>/quant_manifest.json`'s
`layout` field, which describes the *original* tf2onnx export, not the compiled artifact:

| model | original ONNX input | **compiled/deployed input (verified)** | note |
|---|---|---|---|
| `ad-toycar` | `[N,640]` flat | **same**, flat | no spatial dims, no ambiguity |
| `ds-cnn-kws` | `[N,49,10,1]` NHWC | **same** `[N,49,10,1]` | C=1 → NHWC and NCHW are byte-identical anyway |
| `resnet8-cifar10` | `[N,32,32,3]` NHWC, leading `Transpose` to NCHW before the first `Conv` | **`[N,3,32,32]` — NCHW, no Transpose node** | the rewrite folded the input declaration itself to NCHW (`quartus/coredla_hyperram_ed/ip/README.md` §3: "re-exporting the whole graph NCHW-native ... an exact input-layout change, not a numerical one", verified bit-for-bit vs the original by onnxruntime, max abs diff 1.19e-7) |
| `mobilenetv1-025-vww` | `[N,96,96,3]` NHWC, leading `Transpose([0,3,1,2])` | **`[N,3,96,96]` — NCHW, no Transpose node** | same story, README §3: "Remove the leading Transpose and make the graph's own input NCHW", onnxruntime-verified **bit-exact** (max abs diff 0.0) |

Independently re-confirmed in this session (not just re-cited from the README) by loading both
`-rewritten.onnx` files with `onnx.load` and reading `graph.input[0]`'s declared shape plus the
first consuming node — `resnet8-cifar10-rewritten.onnx` and `mobilenetv1-025-vww-rewritten.onnx`
feed `input_1` **directly into the first `Conv`**, no `Transpose` node anywhere upstream of it, shape
`[?,3,H,W]`. Contrast with `models/onnx/<id>.onnx` (the pre-rewrite export): `graph.input[0]` is
`[?,H,W,3]` and the very first node is `Transpose(input_1)` feeding the `Conv`.

**Net finding: `quant_manifest.json`'s `"layout": "NHWC"` field is correct for the *original* model
but wrong for the *compiled* one, for exactly 2 of 4 models** (`resnet8-cifar10`,
`mobilenetv1-025-vww`). `sw/packer/layouts.py`'s existing `nchw`/`nhwc` transforms assume a
CHW-native *source* array (its own docstring), which doesn't match this repo's HWC-native (PIL/
numpy image) datasets either way, so `build_tiny_bundle.py` does the HWC→CHW transpose itself
(`np.transpose(q, (2, 0, 1))`) right before serializing, rather than reusing `layouts.py` as-is.
`ds-cnn-kws` and `ad-toycar` need no transpose (single-channel / flat).

Byte order within a tensor is row-major (numpy/C order) in all cases — nothing here disturbs that.

## 3. The scale/zero-point drift this work found (and fixed)

**Rule enforced by `build_tiny_bundle.py`: never trust a cached manifest — always re-derive
`(scale, zero_point)` live from the *deployed* IR's own input `FakeQuantize` node**
(`quartus/coredla_hyperram_ed/ip/models/<model_id>/<model_id>.xml`, via
`common.fakequantize_params_at_input` + `common.signed_int8_affine_params`, exactly as
`extract_quant_manifest.py` does — just pointed at the deployed IR, not `models/ir/`).

This caught a real bug, live: `models/ir/ad-toycar/quant_manifest.json` had `scale =
0.5054694381414675`, but the deployed IR (`quartus/coredla_hyperram_ed/ip/models/ad-toycar/
ad-toycar.xml`) — confirmed **byte-identical** (sha256) to `models/ir/ad-toycar/int8/ad-toycar.xml`
— actually encodes `scale = 0.5122092452703738` when read fresh. The manifest was simply stale
(written by an earlier `extract_quant_manifest.py` run, before the int8 IR was last regenerated,
and never refreshed). This is `docs/parity_debugging.md` failure mode **#3 ("scale/zero-point
drift")**, caught here rather than shipped: a packer that trusted the manifest would have quantized
every `ad-toycar` input with the wrong scale, and every hardware record would show up as a "parity
mismatch" that isn't a hardware bug at all.

**Fixed as part of this work**: re-ran `sw/model_prep/extract_quant_manifest.py --models ad-toycar`
to refresh the stale manifest in place (now `scale = 0.5122092452703738`, matching the deployed IR
and its correct `int8_ir_sha256`). The other three models' manifests already matched their deployed
IR's fresh-extracted scale (`ds-cnn-kws`, `resnet8-cifar10`) or were numerically identical despite a
differing `.xml`/`.bin` sha256 (`mobilenetv1-025-vww` — see §5), so nothing else needed a fix, but
`build_tiny_bundle.py` re-derives fresh in every case regardless, by design, so this class of bug
can't recur silently.

Fresh values used for every bundle in this delivery (`manifest.json`'s `quant` block per model,
also embedded in `results/ph2_<id>-int8-deployed_*.json`'s config):

| model | scale | zero_point |
|---|---|---|
| `ad-toycar` | 0.5122092452703738 | 0 |
| `ds-cnn-kws` | 0.3418698329551547 | 0 |
| `resnet8-cifar10` | 0.9389804017310049 | -128 |
| `mobilenetv1-025-vww` | 0.0038478533426920575 | -128 |

## 4. What "golden output" means here — and what it deliberately does NOT include

Each record carries:
- **`cpu_pred`** (classifiers only) — the argmax of the **deployed IR's** own OpenVINO
  FakeQuantize-simulated INT8 inference on that exact input (bit-identical input to what hardware
  gets). This is the primary, always-available correctness gate
  (`run_tiny_benchmark.accuracy_crosscheck`'s `cpu_agreement_pct`).
- **`label`** — MLPerf ground truth (class index for the 3 classifiers; 0=normal/1=anomaly, from the
  filename convention, for `ad-toycar`).
- **NOT included: a bit-exact device-INT8 `ref_output` tensor.** `dla_compiler` performs its own
  final-layer INT8 requantization internally when it compiles the network for the FPGA (the deployed
  IR itself keeps the final Dense/Add + Softmax in **float**, matched by direct inspection: no
  output `FakeQuantize` node exists between the last `Add` and `Result` in any of the four deployed
  IRs), and that requantization scale is **not observable off-hardware** — `dla_compiler --help`
  exposes no simulate/reference-dump mode, and no vendor doc in this repo's read set states the
  convention. Fabricating one would violate AGENTS.md ("never invent a number"). This is the honest
  boundary of what's determinable without the board: **argmax/label agreement is the parity gate
  here; true byte-exact output comparison is a hardware-bring-up task**, consistent with
  `run_tiny_benchmark.py` already treating `ref_output_file` as optional and
  `docs/parity_debugging.md`'s existing playbook (case #1, tie-breaking, is the only place raw
  output bytes would add value the argmax gate can't — decode via `sw/packer/inspect_recimg.py`-style
  tooling once real mismatches appear on hardware, not before).
- `ad-toycar` additionally carries `cpu_recon_mse` (float, informational) — the deployed IR's own
  reconstruction MSE for that one frame vector, and `audio_file`/`frame_index` so per-file MLPerf AUC
  can be reassembled later if the on-board harness is extended to run every frame of a file (today's
  `run_tiny_benchmark.py` reports the CPU-INT8 AUC as the reproduced number and gates on
  frame-level correctness only — see its `build_accuracy_result` NOTE).

## 5. Fresh accuracy numbers computed against the *deployed* IR (not `models/ir/`)

`results/ph2_<id>-int8-deployed_<date>.json` (`kind: "reference"`, schema-valid — checked with
`scripts/validate_results.py`) — the full test set, run through the **exact IR that was compiled
into the `.aot`** (`quartus/coredla_hyperram_ed/ip/models/<id>/<id>.xml`), not the
`models/ir/<id>/int8/` copy `eval_int8_cpu.py`'s existing `ph2_<id>-int8_*.json` numbers use. Three
of four models' deployed `.bin`/`.xml` differ (sha256) from the `models/ir` copy — a different NNCF
quantization run happened when the model was rewritten for `dla_compiler` (pool-decompose/NCHW
rewrite, `quartus/coredla_hyperram_ed/ip/README.md` §3) — so the existing `ph2_<id>-int8_*.json`
numbers are a **different, close-but-not-identical INT8 model** than what's on the board:

| model | `models/ir` deployed? (sha256 match) | old `ph2_<id>-int8` | fresh `ph2_<id>-int8-deployed` | Δ |
|---|---|---|---|---|
| `ad-toycar` | **yes** (byte-identical) | AUC 0.7811054378011925 | AUC 0.7811054378011925 | 0 (same file) |
| `ds-cnn-kws` | no | top1 0.9177914110429448 | top1 0.9171779141104295 | -0.0006 |
| `resnet8-cifar10` | no | top1 0.8671 | top1 0.8664 | -0.0007 |
| `mobilenetv1-025-vww` | no | top1 0.8584199963510308 | top1 0.8584199963510308 | 0 (numerically identical despite differing sha256 — same quantization result, different export metadata) |

None of these deltas are alarming (≤0.07 points), but they are **real** and the bundles' `cpu_int8`
field in `reference.json` cites the fresh `-deployed` number, not the older one, so the orchestrator
is always comparing hardware against the model that's actually on the board.

## 6. Regenerating / extending a bundle

```bash
cd sw/model_prep
# small, committed parity set (already done for all 4 models):
../../.venv/bin/python build_tiny_bundle.py --model resnet8-cifar10 --n-records 16 \
    --out ../../results/tiny_bundles/resnet8-cifar10 --write-accuracy-result

# larger / full accuracy set (gitignored; pass the full per-model test-set count for the true
# MLPerf-Tiny full-set number: ad-toycar 2459, ds-cnn-kws 4890, resnet8-cifar10 10000,
# mobilenetv1-025-vww 10962):
../../.venv/bin/python build_tiny_bundle.py --model resnet8-cifar10 --n-records 10000 \
    --out ../../results/tiny_bundles/resnet8-cifar10/accuracy_full
```
`--n-records` picks that many indices **evenly spread** (`np.linspace`, no RNG) over the model's
real MLPerf test split — deterministic and reproducible, not a prefix (which for `ad-toycar` in
particular would otherwise pick only `anomaly_*` files, since they sort alphabetically before
`normal_*`; spreading fixes that — the 16-record parity set is 7 anomaly / 9 normal across all 4
ToyCar machine IDs).

Datasets are read from the already-cached `datasets/` (fetched by `sw/model_prep/models/*.py`'s own
`fetch_dataset`, invoked automatically if missing).

## 7. Unit-tested loader

No new loader was needed — `sw/host/run_tiny_benchmark.py:load_bundle` (already unit-tested,
`sw/host/tests/test_run_tiny_benchmark.py`) reads this exact format. Verified in this session against
all four freshly-built bundles (off-board, no hardware):

```
model            metric  output_bytes  n_records  first-record input_bytes length
ad-toycar        auc     640           16         640
ds-cnn-kws       top1    12            16         490
resnet8-cifar10  top1    10            16         3072
mobilenetv1-025-vww top1 2             16         27648
```
`output_bytes` matches each deployed IR's own output tensor element count (`ad-toycar`: 640-dim
reconstruction; `ds-cnn-kws`: 12 classes; `resnet8-cifar10`: 10 classes; `mobilenetv1-025-vww`: 2
classes), read live from the IR (`deployed_output_bytes` in `build_tiny_bundle.py`) — not hardcoded.

## 8. How the orchestrator feeds a bundle into the driver

Exactly the existing, already-unit-tested path — nothing new to wire up:

```bash
scripts/devkit_lock.sh with "coredla-tiny-agent" "MLPerf Tiny on-board run" -- \
    bash -lc 'source scripts/env.sh && \
        python sw/host/run_tiny_benchmark.py \
            --bundle results/tiny_bundles/ds-cnn-kws \
            --sof <programmed .sof> \
            --arch-file models/arch/AGX3_Performance.arch \
            --fclk-mhz <measured/retuned clk_dla or clk_ddr, per docs/coredla_hyperram_onboard_findings.md §3c> \
            --mode both --out-dir results/'
```
See `docs/tiny_hardware_benchmark_runbook.md` for the full command sequence and result-JSON shape;
this document only fills in the bundle-format gap that runbook's §0.3 left as a TODO. For the
**HyperRAM** path specifically (this branch, `ph3-hyperram-axi4-coredla`) the same
`bundle.records[i].input_bytes` get guard-banded-written into HyperRAM at each model's
`config_base_addr`-relative input offset instead of the DDR-free on-chip path — the record *bytes*
are identical either way; only the transport differs
(`docs/coredla_hyperram_onboard_findings.md` §6).
