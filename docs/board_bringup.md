# Hostless-JTAG example: C-series devkit → AXC3000 bring-up (issue #7)

PLAN §9 PH1: prove the FPGA AI Suite inference IP runs on our silicon before building anything
custom. AI Suite 2026.1.1 ships the design as example ID `agx3c_jtag` ("Agilex 3 Hostless JTAG
Example Design", `dla_build_example_design.py list` / `details agx3c_jtag`), built for the **Altera
Agilex 3C-Series Development Kit** — a different, Intel-branded devkit, not the Arrow AXC3000. This
document records every deviation found while trying to retarget it to the AXC3000, in the order they
were found, and why each one is or isn't a straightforward pin/device swap.

## Step 1 — stock build (unmodified, C-series devkit target)

Baseline per the issue: get a clean stock compile before touching anything.

```
source scripts/env.sh
dla_build_example_design.py build -o <builddir> -f agx3c_jtag \
    $COREDLA_ROOT/example_architectures/AGX3_Small_NoSoftmax.arch --skip-compile
dla_build_example_design.py quartus-compile <builddir>
```

- IP generation: unlicensed (no `LM_LICENSE_FILE`/`ALTERAD_LICENSE_FILE` set) — `dla_create_ip`
  logged "Building unlicensed version", consistent with the 100,000-inference cap
  `docs/toolchain.md` already verified against the upstream README (not the possibly-stale PLAN
  §9 PH1 "10,000" figure).
- Quartus compile ran to completion against the stock project (`top.qsf`, device
  `A3CY135BM16AE6S`, unmodified `pin_assignments.sdc`, `jtag_example.sdc`), including the design's
  own automated `dla_adjust_pll.tcl` fmax-seeking re-tune (build once, measure achieved fmax,
  regenerate the IOPLL for the actually-achievable frequency, then Assembler). Every stage's own
  summary line, in order (`_tmp_stock_build/qcompile.log`, this session, 18 min 6 s total):
  ```

  Fitter placement was successful
  Quartus Prime Fitter was successful. 0 errors, 1 warning
  Quartus Prime Timing Analyzer was successful. 0 errors, 3 warnings   (pre-PLL-adjust STA)
  Quartus Prime Timing Analyzer was successful. 0 errors, 1 warning   (post-adjust STA)
  Quartus Prime Compiler Database Interface was successful. 0 errors, 1 warning
  ```

  Output: `_tmp_stock_build/AGX3_Small_NoSoftmax.sof` (2,612,123 bytes). `dla_compile_report.json`
  (generated by the flow itself): `ip_fmax 414.94 MHz`, `platform 414.81 MHz` (this stock design's
  own automated PLL re-tune target on the C-series devkit's faster E6 speed grade — not an AXC3000
  E7S number, and not comparable to PLAN §2's 250-300 MHz planning point without re-deriving for
  our device); resources `dsp_blocks [32,184]`, `m20ks [199,353]`, `alms [35370,45800]`. This proves
  the toolchain + AI Suite flow end to end before any AXC3000-specific change, satisfying the
  issue's step 1 gate. (The pre-adjust STA's "3 warnings" include Critical Warning 19536 on

- Ran from a `dla_build_example_design.py build --skip-compile` + `quartus-compile` split so the

  above, same policy as `models/ir/`, `.gitignore`), so it isn't part of this PR's diff.

## Step 2 — deltas for AXC3000, in the order found

### 2a. Device (straightforward, needs a re-fit)

Stock: `A3CY135BM16AE6S` (`top.qsf` `DEVICE`, and `example_design.json`'s
`project_info.device_name`). AXC3000: **`A3CY100BM16AE7S`** (Arrow "AXC3000 Evaluation Board: User
Guide" v1.2.1 §2.3.1, and this repo's own `quartus/smoke/smoke.qsf` / `quartus/hyperbus_smoke/`).
Both are Agilex 3 C-series parts but different pin-count/ball-map codes (135 vs 100) and speed
grade (E6 vs E7S) — not a drop-in re-target, a straightforward re-fit against the new device once
the design's I/O list matches real AXC3000 pins (see 2d below).

### 2b. Board string (cosmetic)

Stock: `set_global_assignment -name BOARD "Agilex 3 FPGA C-Series 135B Development Kit"`. No Arrow
AXC3000 board preset ships in Quartus; the `smoke`/`hyperbus_smoke` projects in this repo already
just omit this assignment. Same approach here.

### 2c. Reference clock (tractable, not attempted this session — see "What this session did not attempt" below for why)

Stock: `i_pll_ref_clk`, 3.3 V LVCMOS, board pin (C-series devkit specific), and the design's own
README states "the reference clock frequency of the PLL is 100 MHz" — the IOPLL IPs
(`ed_zero_dla_pll.ip` / `ed_zero_jtag_pll.ip`, both parameterized inside `qsys/ed_zero.tcl`) are
generated assuming a 100 MHz input to hit their fixed 150 MHz (DLA clock) / 100 MHz (JTAG clock)
outputs.

AXC3000: only a fixed **25 MHz** oscillator is available for FPGA fabric clocking (Arrow User Guide
§2.3.3.7 "Oscillators": "25 MHz (for FPGA clocking)"; §3.1.1: `CLK_25M_C` @ pin **A7**, 1.2 V — see
`quartus/constraints/axc3000_board.tcl`). A 25 MHz reference into the same IOPLLs needs their
M/N/C divider parameters regenerated (in `qsys/ed_zero.tcl`'s `dla_pll`/`jtag_pll` component
instantiation) to still land on 150 MHz / 100 MHz outputs, or an explicit decision to accept
different achievable output frequencies. This is ordinary IOPLL re-parameterization — tractable on
its own — but was not attempted this session because completing it produces no closer-to-working
bitstream while 2d (below) remains unresolved.

### 2d. Reset (straightforward)

Stock: `i_fpga_core_resetn`, board-specific pin, active-low, synchronized in `top.sv` via
`dla_cdc_reset_async`. AXC3000: the general-purpose active-low **USER button** (Arrow User Guide
§3.3: "the USER button ... drives the associated signal line to a low logic level [when pressed]
... the user is responsible for adding an internal weak pullup resistor") @ pin **A12**, 1.2 V —
mapped in `quartus/constraints/axc3000_board.tcl`. (There is a second button, S1 @ AG15, wired to
the device's own `nCONFIG`/`RSTn` — a configuration-reset pin, not user fabric logic; not used
here, matching the stock design's single-fabric-reset-input shape.)

### 2e. JTAG (unchanged — confirmed, not assumed)

Both the C-series devkit and the AXC3000 route JTAG to the device's dedicated SDM/JTAG pins through
their own on-board USB-Blaster-class bridge chip (generic "USB Blaster III" on AXC3000 per Arrow
User Guide §3.10.2, driven by a UB3/FT2232H). Confirmed directly, not assumed: the stock design's
own `pin_assignments.sdc` (the *entire* non-JTAG user pin list — LPDDR4 + 2 misc signals) and
`jtag_example.sdc` (only ever references `altera_reserved_tck`/`tms`/`tdi`/`tdo`/`ntrst`, Quartus's
reserved names for the device's dedicated JTAG pins) contain **zero** location assignments for
JTAG signals. Nothing to adapt here — matches the issue's own expectation ("JTAG (unchanged,
dedicated pins)").

### 2f. LPDDR4 global memory — the real blocker, not a pin swap

This is the deviation that matters. Evidence, read directly from the vendor-generated project
(`_tmp_stock_build/hw/`, Step 1), not inferred:

- **`top.sv`'s entire non-JTAG, non-clock/reset port list *is* a 32-bit LPDDR4x32 interface**:
  `io_lpddr4_comp1_dq[31:0]`, 4x `dqs_p`/`dqs_n`, 4x `dmi`, 6-bit `ca`, `ck_p`/`ck_n`, `cs`, `cke`,
  `reset_n`, plus a differential refclk and an `rzq` calibration pin. There is no other external
  memory port on this design.
- `qsys/ed_zero.tcl`'s `instantiate_emif` proc comment: `"Instantiate an LPDDR4x32 interface ...
  LPDDR4 interface is physical bank 2A"`, parameterized `emif_lpddr4_ref_clk_frequency_mhz =
  166.6667`, `emif_lpddr4_phy_freq_mhz = 1066.667` (default) — a real, IO-bank-specific,
  high-speed synchronous DRAM PHY, not a soft/optional block.
- **This wiring is hard, not conditional.** CoreDLA's `dla_platform_wrapper` (instantiated in
  `top.sv`) exposes a fixed AXI4 "DDR" port group (`o_ddr_arvalid/araddr/...`,
  `C_DDR_AXI_ADDR_WIDTH=32`, `C_DDR_AXI_DATA_WIDTH=256`) regardless of which arch file
  (`AGX3_Performance` / `AGX3_Small_NoSoftmax` / `AGX3_Small_Softmax`) generated the IP.
  `ed_zero.tcl`'s `instantiate_emif`/`instantiate_emif_bridge` connect that AXI4 bus straight to
  the LPDDR4 EMIF IP's AXI4 slave (`add_connection ${emif_data_bridge}.m0/${emif}.s0_axi4`) with
  no alternate/optional path.
- **The AXC3000 has no LPDDR4 at all.** Arrow User Guide §2.3.3.2/§2.3.1: its only off-chip RAM is
  a **128 Mb (16 MB) Winbond HyperRAM**, an 8-bit HyperBus DDR part — a different signaling
  protocol, different pin count (12 pins vs. LPDDR4's 40+), different pins entirely (§3.6, and
  `quartus/constraints/axc3000_board.tcl` here). PLAN §1's own board budget table lists HyperRAM,
  QSPI flash and on-chip M20K as the AXC3000's *only* memory tiers — no LPDDR4 line item.
- **No vendor DDR-free fallback exists for Agilex 3 in this AI Suite release.** Checked directly,
  not assumed: `dla_build_example_design.py list` shows DDR-free hostless designs only for Agilex 5
  (`agx5e_modular_ddrfree`) and Agilex 7 (`agx7_iseries_ddrfree`) — **no `agx3*ddrfree` entry**.
  Confirmed again at the architecture-file level: `ls $COREDLA_ROOT/example_architectures/` has
  `AGX7_Streaming_ocp_Ddrfree*.arch` variants but no AGX3 counterpart. PLAN §9 PH1's text ("Also
  the hostless DDR-free example ([HL-NO-DDR])... the 'below the line' configuration verbatim")
  reads as if this exists for Agilex 3 too; in the installed FPGA AI Suite 2026.1.1 it does not.
  Flagging this loudly per AGENTS.md rather than silently treating PH1 as fully deliverable for
  DS-CNN's below-the-line case — **a maintainer should confirm whether a newer AI Suite release
  adds an AGX3 DDR-free example, or whether PLAN §9 PH1 should be reworded to scope DDR-free to
  AGX5/AGX7 only.**

**Conclusion:** retargeting `agx3c_jtag` to the AXC3000 is not "adapt device + pinout" — it requires
replacing the platform's global-memory subsystem (LPDDR4 EMIF → something HyperRAM-backed), which
means hand-editing the vendor-generated `ed_zero.tcl` Platform Designer script to delete the EMIF
instantiation/wiring and substitute a HyperBus-backed (or on-chip-RAM-backed) AXI4/Avalon-MM slave
in its place. That is exactly PLAN §9 PH3's scope ("HyperRAM integration ... Platform Designer
system: JTAG-Avalon master + mSGDMA + HBMC + inference IP"), not PH1.

## What this session did not attempt, and why

This session did **not** hand-edit `ed_zero.tcl` to remove the LPDDR4 EMIF and wire in a
replacement. Reasons, concretely:

1. **The replacement piece doesn't exist in integrable form yet.** The only HyperBus controller in
   this repo (issue #13, `rtl/hyperbus/hbmc_core.sv` + `hyperbus_top.sv`) exposes a 16-bit
   Avalon-MM slave, not an AXI4 slave matching CoreDLA's `C_DDR_AXI_DATA_WIDTH=256` port group, and
   has never been wrapped as a Platform Designer component. It has also never itself been compiled
   through Quartus — issue #13's own PR (#29) explicitly left its Quartus smoke compile as a
   hardware/tooling handoff, since no working Quartus existed in that session. This session is the
   first with a working Quartus + AI Suite toolchain, so that gap is real and current, not stale.
2. **A quick "make it compile" substitute (e.g. tying the DDR AXI4 bus off to constants, or an
   undersized on-chip-RAM stand-in never validated against CoreDLA's actual global-memory
   contract) would satisfy the *letter* of "AXC3000 build compiles with timing met" while being
   functionally meaningless** — inference would never produce a correct result on real hardware.
   AGENTS.md and this issue's own "Do not fake results" instruction rule that out.
3. Given (1) and (2), doing this properly is real, separate, PH3-scale engineering work (a
   Platform-Designer-integrated, AXI4-or-wider HyperBus/on-chip-RAM global memory, validated in
   simulation before a Quartus attempt) — not something to rush through inside this issue just to
   check a box.

**Acceptance criterion 2 ("AXC3000 build compiles with timing met") is therefore left unchecked.**
What *is* delivered towards it: the real board pinout/clock/reset (`axc3000_board.tcl`/`.sdc`,
usable immediately once a design's I/O list matches real AXC3000 pins), and this precise account of
the one piece that blocks a full compile, so a future PH3 issue does not have to rediscover it.

## HyperRAM pin discrepancy (found while writing `axc3000_board.tcl`)

Two independent Arrow sources for the AXC3000's HyperRAM pinout disagree on 2 of 12 pins:

| Signal | Arrow User Guide v1.2.1 §3.6 (2025-06-30, primary used) | `refdes-agilex3` `axc3000_pin_assignment.tcl` (tag `QPDS25.1.1_QPDS_REL_PR`) |
|---|---|---|
| DQ[0:7] | C3, C2, B4, B6, D3, A4, B3, C6 | identical |
| HRESETn | F7 | identical |
| RWDS | A6 | identical |
| **CSn** | **C7** | **D8** |
| **CLK** | **B5** | **D7** |

`axc3000_board.tcl` uses the User Guide's values (the more current, explicitly versioned/dated
official document) but this is flagged, not silently resolved: this session could not independently
adjudicate it against the board schematic (`images/AXC3000/SCH-TEI0131-01-P001.PDF` in
`github.com/ArrowElectronics/Agilex-3`) because that PDF would not rasterize in this environment
(`pdftoppm`/`poppler-utils` unavailable; the shorter User Guide PDF rendered fine via the same tool
path, so this looks like a poppler limit on a larger/more complex vector PDF, not a fetch failure —
the file did download, 1.57 MB, valid PDF header). **Before this pinout is trusted on real
hardware, re-derive CSn/CLK from the schematic PDF directly (or from a multimeter continuity check
on the board) and update `axc3000_board.tcl`'s comment + assignments if the schematic disagrees
with the User Guide.**

## HyperRAM voltage discrepancy against PLAN §1 (found the same way)

PLAN §1's board budget table reads: `HyperRAM | 128 Mb = 16 MB | Winbond W957D8NB, ×8 HyperBus DDR,
**1.8 V**`. Arrow's User Guide v1.2.1 (§2.3.3.2, §3.6, and its Figure 3-7 wiring diagram caption)
states the AXC3000's HyperRAM is **1.2 V, 250 MHz**, and its per-pin table lists I/O standard
"1.2V" for every HyperRAM signal (DQ0-7, RWDS, CLK, CSn, HRESETn) — consistent with §2.3.3.6's bank
voltage table ("Bank 3A: 1.2 V", the bank the HyperRAM pins listed in §3.6 sit in). `PLAN.md`'s own
preamble says fix-and-flag if its numbers disagree with a source; per AGENTS.md ("Change
`docs/PLAN.md` numbers without flagging it loudly" is one of the "must never do" items) this is
flagged here rather than silently edited: **PLAN §1's "1.8 V" for the HyperRAM looks stale/wrong
against the current, versioned, official Arrow board documentation, which says 1.2 V** — a
maintainer should confirm and correct PLAN.md. `axc3000_board.tcl` here uses **1.2 V** (the Arrow
User Guide value) for all HyperRAM pins, since it is describing the real board's actual pins, not
PLAN.md's number. This also means issue #13's `hyperbus_smoke.qsf` placeholder comment ("HyperRAM
is 1.8 V") was carrying the same stale assumption — not fixed here since #13 is a different,
already-merged issue's file, but worth the same maintainer follow-up.

**Update (doc review pass):** `docs/PLAN.md` §1 has since been corrected to **1.2 V**, citing this
section. `quartus/hyperbus_smoke/hyperbus_smoke.qsf`'s placeholder comment still says 1.8 V and has
not been touched (out of scope for a docs-only pass) — flagged again here for whoever next edits
that file.

## Clock I/O-standard discrepancy (same two sources)

`CLK_25M_C` @ pin A7: Arrow User Guide v1.2.1 §3.1.1 table says I/O standard **"1.2V"**;
`refdes-agilex3`'s `axc3000_pin_assignment.tcl` asserts **"1.3-V LVCMOS"** with an inline comment
("... used to fake Quartus since [Quartus lacks an exact standard for this rail]"). `axc3000_board.tcl`
uses 1.2 V (User Guide) as primary and documents the 1.3-V LVCMOS fallback inline, per the same
reasoning as the CSn/CLK note above — re-verify empirically (Fitter acceptance / board bring-up)
before relying on either.

## `axc3000_board.tcl`/`.sdc` verified against the real device (this session)

Built a throwaway single-file design (`pintest.sv`, ports named exactly `CLK_25M_C`, `USER_BTN`,
`RLED`/`GLED`/`BLED`/`LED1`, `hb_dq[7:0]`, `hb_rwds`, `hb_cs_n`, `hb_ck`, `hb_rst_n`) against device
`A3CY100BM16AE7S`, sourcing `axc3000_board.tcl` + `axc3000_board.sdc` verbatim, and ran it through
`quartus_syn` then `quartus_fit`:

```
Quartus Prime Synthesis was successful. 0 errors, 15 warnings
Quartus Prime Fitter was successful. 0 errors, 1 warning
```

No illegal-pin or IO-standard-conflict errors on the real device — every location/IO-standard
assignment in `axc3000_board.tcl` is at least legal for `A3CY100BM16AE7S`. (The synthesis warnings
are expected/cosmetic for a throwaway tie-off design: stuck-at pins, disabled tri-state buffers on
an unused `hb_dq`/`hb_rwds` bidir with no real controller behind it, and the standard Agilex 3
Reset-Release-IP advisory — none are pin-legality problems.) This is **not** a claim that the
CSn/CLK pin values or either voltage-standard discrepancy above are electrically correct on the
real board — only that the Tcl runs clean and the pins/standards it asserts are ones Quartus
accepts for this device. Scratch project deleted after the check, not committed.

- `quartus/constraints/axc3000_board.tcl` — real AXC3000 pinout (clock, reset, LEDs, HyperRAM),
  sourced from the two provenance documents above, both discrepancies flagged inline.
- `quartus/constraints/axc3000_board.sdc` — base SDC for the board's 25 MHz input clock + reset.
- `sw/host/smoke_infer.py` — System Console push-one-tensor/read-one-tensor-back script against the
  stock design's actual JTAG-Avalon memory map (verified from the AI Suite 2026.1.1 runtime source,
  not guessed — see the script's own docstring); the CoreDLA CSR start/done handshake itself is left
  `NotImplementedError` (undocumented internal vendor protocol, would need reverse-engineering
  during real board bring-up).
- This document.
