# Instructions for AI implementers

You are implementing one GitHub issue at a time in this repository. The issues are the roadmap; each
one is written to be self-contained. Follow this file plus the issue body exactly.

## Before you start any issue

1. Read `docs/PLAN.md` — it is the canonical source of every number in this project. Never invent a
   number; cite the PLAN section instead.
2. Read the issue top to bottom, including **Depends on**. If a dependency issue is not closed, stop
   and say so instead of guessing at its outputs.
3. Look at the directory README for every path the issue tells you to touch.

## Workflow

- One issue = one branch = one PR. Branch name: `issue-<number>-<short-slug>` (e.g. `issue-15-scoreboard-rtl`).
- Commit early, small, and with messages that name the issue: `scoreboard: add latency histogram (#15)`.
- The PR description must walk through each **Acceptance criteria** checkbox and state how it was
  verified (command + output, or "hardware-gated, see below").
- If an acceptance criterion needs the physical AXC3000 board, a Quartus install, or a human with a
  meter, implement everything up to that point, check off what you verified, and list the remaining
  steps under a "## Hardware handoff" section in the PR. Do not fake results.
- Never mark a checkbox you did not verify. "It should work" is not verification.

## RTL conventions (`rtl/`)

- SystemVerilog-2017, one module per file, filename = module name.
- Follow the Hyperflex disciplines from PLAN §3 LV1 in ALL datapath code: reset-less pipeline
  registers on the datapath; synchronous reset only for architectural state (FSMs, counters, CSRs);
  no clock gating; no asynchronous resets anywhere.
- Clock-domain crossings only via the wrappers in `rtl/common/` (to be created by the issues that
  need them). Never hand-roll a synchronizer inside a functional module.
- Every module that contains logic (not just wiring) gets a self-checking testbench in `sim/`.
  Testbenches must run under **Verilator** (`verilator --binary`) when the module uses no device
  primitives; device-primitive code (DSP/M20K/IO instantiations) is instead validated by the Quartus
  fitter report, and the issue says which reports to check.
- Avalon-MM is the on-chip bus everywhere (Platform Designer ecosystem). Signal naming:
  `csr_address`, `csr_read`, `csr_readdata`, `csr_write`, `csr_writedata`, `csr_waitrequest`.
- Parameterize widths/depths; no magic numbers — put shared constants in
  `rtl/common/bench_pkg.sv` as they appear.

## Python conventions (`sw/`, `scripts/`)

- Python ≥3.10, standard library + numpy by default; heavier deps (openvino, nncf, torch,
  tensorflow) only in the tool that needs them, pinned in that tool's own `requirements.txt`.
- Every tool is an argparse CLI with `--help` text and a module docstring stating which issue it
  implements. No notebooks.
- Deterministic: fixed seeds, sorted file iteration, versions recorded in output manifests.
- Unit tests with pytest next to the code (`test_*.py`); pure-logic parts (packing, quantization
  math, report parsing) must be tested without needing Quartus, OpenVINO, or hardware — isolate
  those behind small functions.

## Results (`results/`)

- Every measured or estimated number lands as one JSON file conforming to
  `results/schema/result.schema.json`. PLAN §10: **numbers without configs are noise** — the schema
  makes the config fields mandatory. Estimates set `"kind": "estimate"`, measurements
  `"kind": "measured"`; never overwrite a measurement with an estimate.

## Quartus (`quartus/`)

- One revision per experiment, scripted: every project must be reproducible by a single
  `scripts/build.sh <target>` invocation using `quartus_sh --flow compile` — no state that exists
  only in the GUI.
- Timing constraints in `quartus/constraints/`, one `.sdc` per clock architecture, commented.
- After every compile, run the tensor-mode audit (issue for `scripts/audit_tensor_mode.py`):
  DSPs silently falling back to classic 18×19 mode are a 10× loss (PLAN §3 LV2).

## Things you must never do

- Invent, extrapolate, or "reasonably assume" a measured number. Estimates are labeled estimates.
- Use JTAG as a data plane in anything performance-measured (PLAN §8 method E).
- Put weights, datasets, or build outputs in git — see `.gitignore`; artifacts are regenerated by
  scripts or fetched by `sw/model_prep/`.
- Change `docs/PLAN.md` numbers without flagging it loudly in the PR description.
